# 1. Introduction

## 1.1 Purpose
The purpose of this Software Requirements Specification (SRS) is to detail the functional and non-functional requirements for the Retail Management System (RMS). This document serves as a formal agreement between the stakeholders and the development team, ensuring that the final product meets the business needs of the retail operations, covering Point of Sale (POS), Inventory Management, and basic Reporting.

## 1.2 Scope
The RMS will manage the core operations of a retail store, including:
1. **Point of Sale (POS):** Handling sales transactions, returns, and discounts.
2. **Inventory Management:** Tracking stock levels, managing suppliers, and receiving goods.
3. **User Management:** Controlling access levels for different roles (Cashier, Manager, Admin).
4. **Reporting:** Generating sales summaries, low-stock alerts, and profit reports.

**Out of Scope:** E-commerce integration, advanced accounting/payroll, and loyalty program management (these may be considered for future phases).

# 2. Overall Description

## 2.1 Product Perspective
The RMS is a standalone, integrated system designed to replace manual and disparate tracking methods. It interacts with hardware peripherals such as barcode scanners, receipt printers, and cash drawers.

## 2.2 User Characteristics
| User Role | Description | Key Tasks | Technical Proficiency |
| :--- | :--- | :--- | :--- |
| **Cashier** | Handles daily sales transactions. | Scanning items, processing payments, handling returns. | Low to Medium |
| **Store Manager** | Oversees daily operations, manages staff, handles discrepancies. | Adjusting inventory counts, running end-of-day reports, managing user accounts. | Medium |
| **Administrator** | Configures system settings, manages suppliers, sets pricing rules. | Adding/editing product catalogs, setting tax rates, managing user roles. | High |

## 2.3 General Constraints
1. **Technology Stack:** The system must be developed using a modern, scalable framework (e.g., Python/Django or Java/Spring Boot).
2. **Hardware Compatibility:** Must support standard USB barcode scanners and receipt printers.
3. **Security:** All sensitive data (payment details, passwords) must be encrypted both in transit and at rest.
4. **Compliance:** Must comply with local tax regulations (e.g., VAT/GST handling).

# 3. Specific Requirements

## 3.1 Functional Requirements (FR)

### FR 1.0: Point of Sale (POS) Module
* **FR 1.1: Item Scanning:** The system shall allow cashiers to scan SKUs to add items to the current transaction list.
* **FR 1.2: Manual Entry:** The system shall allow cashiers to manually search and add items using SKU or product name.
* **FR 1.3: Price Calculation:** The system shall automatically calculate the subtotal, applicable taxes, and grand total for the transaction.
* **FR 1.4: Discount Application:** The system shall support applying percentage-based or fixed-amount discounts, with manager override capability.
* **FR 1.5: Payment Processing:** The system shall accept payments via Cash, Card (simulated/integrated), and other specified methods, calculating change due.
* **FR 1.6: Returns/Refunds:** The system shall process returns by scanning the original item(s) and automatically adjusting inventory and issuing a refund receipt.

### FR 2.0: Inventory Management Module
* **FR 2.1: Product Catalog Management:** Admins shall be able to Create, Read, Update, and Delete (CRUD) product records, including SKU, name, description, cost price, selling price, and category.
* **FR 2.2: Stock Level Tracking:** The system shall automatically decrement stock levels upon sale and increment upon receiving new stock.
* **FR 2.3: Stock Receiving:** Managers shall be able to record incoming shipments, updating stock levels and linking the receipt to a specific supplier invoice.
* **FR 2.4: Low Stock Alerts:** The system shall generate a report/dashboard alert when any SKU falls below a predefined minimum threshold.
* **FR 2.5: Stock Adjustment:** Managers shall be able to perform manual stock adjustments (e.g., due to damage or count discrepancies), requiring a mandatory reason code.

### FR 3.0: User and Security Module
* **FR 3.1: Role-Based Access Control (RBAC):** The system shall enforce access based on user roles (Cashier, Manager, Admin).
* **FR 3.2: Password Management:** The system shall enforce strong password policies (minimum length, complexity).
* **FR 3.3: Audit Logging:** The system shall log all critical actions (e.g., price changes, stock adjustments, voids) including User ID, Timestamp, and Action taken.

### FR 4.0: Reporting Module
* **FR 4.1: Daily Sales Report:** The system shall generate a report summarizing total sales, total items sold, and total revenue for a specified date range.
* **FR 4.2: Top Selling Items Report:** The system shall report the top N best-selling SKUs based on quantity sold over a period.
* **FR 4.3: Low Stock Report:** The system shall generate a comprehensive list of all SKUs below their minimum threshold.

## 3.2 Non-Functional Requirements (NFR)

### NFR 1.0: Performance
* **NFR 1.1: Response Time (POS):** The POS interface must process a standard transaction (up to 20 items) and print the receipt within 5 seconds under normal load.
* **NFR 1.2: Scalability:** The system must support up to 50 concurrent users without significant degradation in performance.

### NFR 2.0: Security
* **NFR 2.1: Data Encryption:** All passwords and sensitive financial data must be encrypted using industry-standard algorithms (e.g., AES-256).
* **NFR 2.2: Session Timeout:** User sessions must automatically time out after 30 minutes of inactivity.

### NFR 3.0: Usability
* **NFR 3.1: Intuitive Interface:** The POS interface must be highly intuitive, requiring minimal training for cashiers.
* **NFR 3.2: Error Handling:** The system must provide clear, actionable error messages to the user, guiding them on how to resolve the issue.

### NFR 4.0: Reliability
* **NFR 4.1: Data Backup:** The system must support automated, scheduled daily backups of the entire database.
* **NFR 4.2: Downtime:** The system must achieve 99.5% uptime during operational hours (7 AM - 10 PM).

# 4. Data Model (Conceptual)

The system will revolve around the following core entities:

* **User:** (UserID, Username, PasswordHash, RoleID)
* **Product:** (SKU, Name, Description, CategoryID, CostPrice, SellingPrice, MinStockLevel)
* **Category:** (CategoryID, Name)
* **Transaction:** (TransactionID, Date, TotalAmount, PaymentMethod)
* **TransactionItem:** (TransactionItemID, TransactionID, SKU, QuantitySold, UnitPriceAtSale)
* **InventoryLog:** (LogID, SKU, ChangeQuantity, ChangeReason, Timestamp)
* **Supplier:** (SupplierID, Name, ContactInfo)
